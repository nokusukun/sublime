# Sublime — TypeScript

TypeScript's power is the type system; most slop is generated by escaping it. Sublime for TypeScript keeps you inside the types, shrinks the surface you cast through, and refuses the ceremonies the model reaches for when it doesn't know what it's modelling.

---

## Context Gathering Protocol (TypeScript)

TypeScript code is only as good as the configuration it compiles under. The same source is rigorous under `strict: true` and meaningless under `noImplicitAny: false`. You MUST resolve the configuration before you write code.

**Required context — read from the project, then confirm with the author:**

- **`tsconfig.json` strictness.** Check `strict`, `noImplicitAny`, `strictNullChecks`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`. If `strict` is off, note it and ask whether you're allowed to turn it on for new files or must match the laxness of the surrounding code.
- **Framework and runtime.** React (SPA) vs Next.js App Router vs Next.js Pages Router vs Node service vs Deno vs Bun vs Cloudflare Workers. These have incompatible module systems, differing `"use client"` rules, and different idioms for data fetching.
- **Package manager.** npm, pnpm, yarn, or bun. Affects lockfile discipline, workspace conventions, and install commands you suggest.
- **Module system.** ESM (`"type": "module"`, `.mjs`, `import`) vs CJS (`require`, `module.exports`). Mixing them is a common source of subtle runtime failure. Check `package.json` and `tsconfig.json`'s `module` and `moduleResolution`.
- **Build and bundle target.** `tsc`, `tsup`, `esbuild`, Vite, Next's built-in bundler, or Bun. Determines what syntax you can emit.

**Gathering order:**
1. If loaded instructions already contain a **Code Context** section, proceed.
2. Otherwise read `.sublime.md` at the project root.
3. If neither exists and the work is non-trivial, run `sublime teach` before writing substantial code. For one-line edits, infer from the surrounding file and state your assumptions at the top of the response.

**CRITICAL:** The surrounding code tells you what the codebase will *compile*. Only the author can tell you what it must keep *doing*. Do not silently upgrade strictness, change `module`, or flip `"use client"` boundaries without asking.

## TypeScript Direction

Before writing non-trivial TypeScript, commit to a posture:

- **Strictness posture.** `strict: true` with `noUncheckedIndexedAccess` is the default. If the project is looser, match the project for edits and propose tightening in a separate diff. Do not mix strictness mid-file.
- **Compatibility constraints.** What Node version? What browsers? What TypeScript version? These decide which syntax (`using`, `satisfies`, `const` type params, `Array.prototype.findLast`) is safe.
- **Consumer surface.** A library's `.d.ts` output is its contract; an internal module's types can be as tight as you like. Library code demands more conservative exports and explicit return types; internal code can lean on inference.
- **The one thing.** What should a reader notice first — that the types make illegal states unrepresentable, that the module boundary is minimal, that the React component does one thing? Pick before you write.

Match the intensity of type craft to the posture. A throwaway script does not need branded IDs. A public SDK cannot afford `any` at its edges.

## Craft Guidelines

### Strictness and configuration

→ Consult [typescript/references/strictness.md](references/strictness.md) for the full flag-by-flag discussion and migration paths from loose configurations.

The type system is load-bearing. Every flag you disable is a category of bug the compiler will no longer find for you. `noImplicitAny: false` alone erases most of what makes TypeScript worth using.

**Always apply these — no reference needed:**

- `strict: true` on new projects. Non-negotiable.
- `noUncheckedIndexedAccess: true` unless you have a specific reason otherwise — array and record accesses should return `T | undefined`, not `T`.
- `exactOptionalPropertyTypes: true` for libraries where `{ x?: number }` meaning "x may be missing" must not collapse with `{ x: number | undefined }`.
- Match the project's existing `tsconfig`. Do not flip flags in a feature PR.

**DO** treat a type error as a real error. Fix the types, not the surface.
**DO** narrow at the boundary and trust the type inside the module.
**DO** use `satisfies` when you want a literal to be checked against a type without widening.

**DO NOT** set `noImplicitAny: false` to make errors go away.
  — You are telling the compiler to stop working for you. Every untyped parameter is now a silent `any` that infects its callers.
**DO NOT** sprinkle `// @ts-ignore` to silence errors. Those are the errors that matter most.
**DO NOT** use `any` to satisfy a generic constraint you do not understand. Read the constraint.

### `any`, `unknown`, and `as`

→ Consult [typescript/references/any-unknown-as.md](references/any-unknown-as.md) for the full discipline, including when each escape hatch is legitimate.

`any` disables the type checker for a value and everything it touches. `unknown` keeps the checker on but forces you to narrow. `as` is an unchecked assertion that this value is what you claim. Each has a narrow legitimate use and a wide slop use.

**Always apply these — no reference needed:**

- `any` is forbidden in production type signatures. If you must, use `unknown` and narrow.
- `as` is for when the compiler cannot prove what you know — JSON parsing that's already been validated, DOM queries you know the shape of, test fixtures. Never to silence an inconvenient error.
- `as unknown as Foo` is a smell on sight. It means "I am lying twice to the compiler." There is almost always a better shape.

**DO** model unknown inputs as `unknown`, then narrow with a type guard or schema parser (Zod, Valibot, Effect Schema).
**DO** use `satisfies` to keep literal inference where assignment would widen it.
**DO** leave a one-line comment when you cast — "parsed upstream by X, see Y" — so the next reader knows it was deliberate.

**DO NOT** write `data as any` to silence an error.
  — You are hiding the bug the type system was trying to show you.
**DO NOT** write `as unknown as Foo` to defeat a structural mismatch.
  — The mismatch is the bug. Fix the type or parse the value.
**DO NOT** type parameters as `any` to "accept anything." Use `unknown` and let the caller prove its shape.

### Discriminated unions and narrowing

→ Consult [typescript/references/unions-and-narrowing.md](references/unions-and-narrowing.md) for tagged-union patterns, user-defined type guards, and exhaustiveness.

Most "half-filled record" types are two types pretending to be one. Model state with a discriminant and the compiler will keep your branches honest.

**Always apply these — no reference needed:**

- Use a tag field (`kind`, `type`, `status`) as the discriminant. Switch on the tag; the compiler narrows the other fields automatically.
- Make illegal combinations unrepresentable. Do not use `error: string | null` alongside `data: T | null` with a silent "one is set" invariant — express it as `{ kind: "ok"; data: T } | { kind: "err"; error: string }`.
- Use `never` in the `default` of an exhaustive switch to make the compiler shout when a new variant is added.

**DO** parse at the boundary so the inside of your module sees only the narrow types.
**DO** prefer user-defined type guards (`function isUser(x: unknown): x is User`) to inline `typeof` chains when the predicate is reused.
**DO** model async state as `{ status: "idle" | "loading" | "ok" | "err" } & { ... }` rather than four nullable fields.

**DO NOT** model a union as a type with optional fields.
  — `{ a?: string; b?: number }` has four inhabitants instead of two. The compiler cannot tell you which branch you're in.
**DO NOT** reach for `as` in a switch branch when narrowing would do it.
**DO NOT** stringly-type a finite set of states. `"pending" | "done" | "error"` is a type; `string` is not.

### Branded and opaque types

→ Consult [typescript/references/branded-types.md](references/branded-types.md) for the full brand pattern, opaque-type techniques, and when to reach for them.

TypeScript is structural. `UserId` and `OrderId` are both `string` and are mutually assignable unless you brand them. If your system has many IDs that must not be swapped, branding is cheap insurance.

**Always apply these — no reference needed:**

- Brand identifiers that share a runtime type but a distinct domain meaning. `UserId`, `OrderId`, `Email`, `ISODateString`.
- Brand values that have been validated — `ValidatedEmail`, `NonEmptyArray<T>` — so the type witnesses the validation.
- Keep the brand one layer thick. Deeply nested nominal wrappers become their own slop.

**DO** centralize the brand constructor in the module that owns the type. The only way to mint a `UserId` is via that module.
**DO** pay the branding cost where an ID-swap bug would be expensive — payments, auth, tenant isolation.

**DO NOT** brand every string. The reader tax is real; spend it where it earns.
**DO NOT** smuggle the brand out through a generic `as Brand<T>` helper that any caller can invoke. That defeats the point.

### Module boundaries and exports

→ Consult [typescript/references/module-boundaries.md](references/module-boundaries.md) for barrel-file consequences, `import type`, and declaration-file hygiene.

The hardest thing to change is your public surface. Keep it small; keep the internal surface smaller.

**Always apply these — no reference needed:**

- Export only what callers need. Everything else stays unexported.
- Use `import type` for type-only imports. It drops cleanly at build time and avoids accidental runtime-side effects.
- Prefer named exports over default. Named exports are grep-able and refactor cleanly; defaults invite divergent rename across the codebase.
- Do not create barrel `index.ts` files that re-export a whole directory. They break tree-shaking and create import cycles.

**DO** let files be named by what they contain — `retries.ts`, `ids.ts`, `formatting.ts`. A `utils.ts` in a TypeScript project is the same dumping ground as in any language, now with type pollution.
**DO** co-locate the type with the value that produces it when there's one producer.
**DO** separate public (`index.ts` or `mod.ts`) from internal (`_internal/` or files prefixed with `_`) deliberately.

**DO NOT** re-export types through three layers of barrel.
  — Each layer is a point where the contract can silently widen.
**DO NOT** write declaration files by hand for code you own. Let `tsc --declaration` emit them.
**DO NOT** export a mutable object. Export a function or a frozen value.

### React patterns

→ Consult [typescript/references/react-patterns.md](references/react-patterns.md) for props discipline, hook restraint, and Next.js App Router specifics.

React's type ergonomics are good; most React-TS slop is cargo from older patterns or from prompting the model into defensive reflexes.

**Always apply these — no reference needed:**

- Type props as an explicit `interface` or `type` at the top of the file. Inline prop types are harder to grep.
- Do not use `React.FC` unless you need its quirks. It adds an implicit `children` you may not want and bakes in a default generic. Write `function Button(props: ButtonProps)`.
- `useEffect` is for synchronizing with external systems — DOM, network, subscriptions. It is not for "running code after render." If you can derive the value, derive it; don't effect-and-set-state.
- `useMemo` and `useCallback` are for measured performance problems. On a primitive comparison, they cost more than they save.
- In Next.js App Router, `"use client"` is a boundary, not a sprinkle. Put it at the leaf that actually needs interactivity; let the parents render on the server.

**DO** fetch data in server components (App Router) or route loaders, not in `useEffect`.
**DO** pass the narrowest prop type that works. A component that takes `{ user: User }` when it only reads `user.name` is leaky.
**DO** let React derive; only effect what React cannot derive.

**DO NOT** put `"use client"` at the top of every file in a Next.js App Router project.
  — You've turned a server-first framework back into a client-first one. You lose streaming, RSC payloads, and half the perf story.
**DO NOT** wrap a primitive comparison in `useMemo`. The comparison is already `O(1)`; the memoization adds a closure and a deps array.
**DO NOT** put objects in `useEffect` deps arrays. Reference equality will fire the effect every render — this is the category of bug that took down the Cloudflare dashboard.
**DO NOT** write `<T extends unknown>` as a workaround for JSX's `<T>` parse ambiguity in `.tsx`. Write `<T,>` with a trailing comma, or move the generic to a non-`.tsx` file.

### Generics restraint

Generics earn their keep when a function's shape is genuinely polymorphic and a concrete type would force duplication. They cost their keep the moment a reader has to trace `<T extends K, K extends keyof U>` to understand what goes in.

Write the concrete version first. Add the generic when the second caller, with a different concrete type, actually appears. "Premature generic" is a standard TS-slop tell — a function signature uses three type parameters and exactly one call site instantiates it with `number`, `string`, `User`.

**DO** inline until a second, genuinely-different caller demands abstraction.
**DO** constrain generics as narrowly as the problem allows — `<T extends Record<string, unknown>>` is more honest than `<T>`.
**DO** let inference do the work. A caller shouldn't have to pass the type parameter explicitly except in unusual cases.

**DO NOT** put a generic on every function signature out of reflex. Most functions take concrete types and return concrete types.
**DO NOT** use a generic to re-implement `any`. `<T>(x: T) => T` that doesn't constrain `T` is expensive identity.

## Hard BANs (TypeScript)

**BAN 1: `any` in production type signatures**
- PATTERN: `function handle(x: any): any { ... }`, `const data: any = ...`, `Array<any>`
- FORBIDDEN: `any` appearing on an exported function, a class field, a public type, or an interop surface
- WHY: `any` disables the checker for the value and infects every caller. It is the single most common way TypeScript codebases rot.
- REWRITE: use `unknown` and narrow with a type guard or schema parser; or write the real type; or in the rare interop case, scope the `any` to the smallest possible expression and comment why.

**BAN 2: `as unknown as Foo` double-cast**
- PATTERN: `const user = payload as unknown as User`
- FORBIDDEN: any double-cast that launders a structural mismatch
- WHY: you are telling the compiler twice that you are lying. The mismatch the double-cast hides is almost always a real bug.
- REWRITE: parse `payload` through a schema and let the parser produce a `User`; or fix the source type; or write a user-defined type guard that actually checks the shape.

**BAN 3: `// @ts-ignore` / `// @ts-expect-error` without a reason**
- PATTERN: `// @ts-ignore` on the line above a type error, no comment explaining why
- FORBIDDEN: any suppression comment without an adjacent sentence explaining the mismatch, the ticket, or the plan
- WHY: the error being suppressed is exactly the kind the compiler exists to show you. Without a reason, the next reader cannot tell if it's load-bearing or a leftover.
- REWRITE: fix the type. If suppression is genuinely required, use `// @ts-expect-error — <reason, ticket ref>` so the comment must justify itself and breaks when the underlying error goes away.

**BAN 4: Optional-chaining cascades on non-nullable types**
- PATTERN: `user?.profile?.name?.trim?.()?.toLowerCase?.()` where `user: User` is already non-null
- FORBIDDEN: `?.` on any value the type system already guarantees is present
- WHY: the chain lies about the contract. Readers learn to ignore `?.`, which then hides the cases where it actually matters.
- REWRITE: trust the type. `user.profile.name.trim().toLowerCase()`. If the value really can be absent, narrow it once with a type guard or early return.

**BAN 5: Empty types**
- PATTERN: `interface Foo {}`, `type Foo = {}`, `interface Marker extends Base {}` with no added members
- FORBIDDEN: any empty object type that isn't a deliberate "anything non-nullish" (and even that is usually wrong — use `object` or `Record<string, unknown>`)
- WHY: `{}` in TypeScript means "any non-nullish value," which is almost never what the writer intended. Empty markers are ceremony.
- REWRITE: delete the type and use its parent directly; or give it real members; or use `Record<string, never>` if you really mean "an object with no properties."

**BAN 6: Enum where a union of string literals would do**
- PATTERN: `enum Status { Pending, Done, Error }` for three values used as tags
- FORBIDDEN: numeric or string enums where a discriminated union would serialize, narrow, and read better
- WHY: TS enums emit runtime objects, don't serialize cleanly, break tree-shaking, and interact poorly with narrowing. Union-of-literals is cheaper, narrower, and JSON-compatible.
- REWRITE: `type Status = "pending" | "done" | "error"`. If you need the value list at runtime, derive it: `const STATUSES = ["pending", "done", "error"] as const; type Status = (typeof STATUSES)[number];`.

**BAN 7: `Promise<void>` wrapper where `async/await` would do**
- PATTERN: `function run(): Promise<void> { return new Promise((resolve) => { doThing(); resolve(); }); }`
- FORBIDDEN: hand-constructed `Promise` wrappers around synchronous or already-async work
- WHY: `new Promise((resolve) => ...)` is error-prone — forgotten rejects, swallowed throws, and a reader tax for no gain. `async` functions return `Promise<T>` automatically.
- REWRITE: `async function run(): Promise<void> { await doThing(); }`. Use `new Promise` only when bridging a callback or event-based API that genuinely isn't promise-shaped.

## The AI Slop Test (TypeScript)

If an experienced TypeScript engineer read this diff and said "an AI wrote this," would they be right on sight? These are the tells:

- Triple `as` cast chains — `value as unknown as SomeType as Readonly<SomeType>`.
- A generic on every function signature, used once with a concrete type.
- `any[]` as the return type of a parse, fetch, or deserialize function.
- `useMemo` wrapping a primitive comparison or a two-field object literal.
- `"use client"` at the top of every file in a Next.js App Router project.
- A four-entry enum where a string union would do.
- `React.FC<Props>` on a component that never reads `children`.
- `interface IUser` paired with `class UserImpl`. Neither prefix belongs in TypeScript.
- `<T extends unknown>` in a `.tsx` file as a workaround for the generic-vs-JSX parse ambiguity.
- Data fetched inside `useEffect` in an App Router project that has server components available.
- A single Tailwind `className` string 180 characters long with no `clsx`, no extraction, no conditional grouping.
- JSDoc re-stating parameter types the TypeScript signature already conveys.
- Optional chaining on values typed as non-null one line above.
- `Promise<void>` returned from a function body that's already `async`.

Good TypeScript is distinctive to its domain. Slop TypeScript is distinctive to the model that wrote it.

## Implementation Principles (TypeScript)

Types are load-bearing. Don't escape them with `as` or `any` — tighten them. Don't over-abstract with generics — inline until a second caller earns them. Parse at the boundary so the inside of your module can trust its inputs, and keep escape hatches scoped, commented, and rare. The compiler is your partner, not an obstacle; when it complains, the right move is almost always to listen, not to silence.
